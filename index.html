part 01

Skills 的由来

从专用 Agent 到通用 Agent 的范式转换

好，我们正式进入第一部分——Skills 的由来。

要理解 Skills 为什么会出现，我们得先回到 AI Agent 的演进历程，看看从"专用"到"通用"这个范式转换中，到底缺了什么。

// part 01 · 演进

AI Agent 的三个时代

01
专用 Agent 时代（2023 以前）

Research Agent、Coding Agent、Finance Agent……每个 Agent 有自己的脚手架、自己的工具集、自己的知识。像 瑞士军刀——每把刀只能做一件事，但做得还行。

02
通用 Agent 时代（2024–2025）

Claude、GPT-4、Gemini——一个模型搞定一切。代码即通用接口，bash + 文件系统就是脚手架。但通用 = 没有专长，缺的是上下文和领域专知。

03
技能增强 Agent 时代（2025–至今）

通用 Agent + 可插拔的技能模块 = 既通用又专业。Skills 提供按需加载的过程性知识和团队上下文，补齐了最后一公里。

AI Agent 经历了三个时代。

第一个时代是专用 Agent。2023 年以前，我们看到的是 Research Agent、Coding Agent、Finance Agent 这种分工很细的 AI。每个 Agent 有自己的脚手架、自己的工具集。像瑞士军刀——每把刀只能做一件事，但做得还行。

第二个时代是通用 Agent。2024 到 2025 年，大模型能力上来了，Claude、GPT-4、Gemini 这些一个模型搞定一切。代码就是通用接口，bash 加文件系统就是脚手架。但问题也来了——通用等于没有专长。你让它分析营销数据，它不知道你的预算规则；你让它写报告，它不知道你们公司的格式要求。

第三个时代就是我们现在看到的——技能增强 Agent。通用 Agent 加上可插拔的技能模块，既通用又专业。Skills 提供了按需加载的过程性知识和团队上下文，补齐了从"懂"到"做好"的最后一公里。

这个范式的转换，就是 Skills 出现的根本原因。

// part 01 · 核心缺口

通用 Agent 缺什么？

🧠 领域专知

品牌规范与模板、法律审查流程、数据分析方法论——这些隐性知识不在训练数据里。

🔄 可重复工作流

每周营销复盘、客户 call 前的准备流程、季度业务评审——流程不是知识，是肌肉记忆。

🛠 新能力

创建 PPT、生成 Excel 报告、构建 MCP 服务器——能力可以被"教会"，而不是等下一个模型版本。

所以通用 Agent 到底缺什么？我把它归纳为三个维度。

第一个是领域专知。品牌规范和法律审查流程这种东西，叫"隐性知识"——它不在公开的训练数据里，而是藏在公司的 wiki、流程文档、老员工的脑子里。你没法指望一个通用模型知道你们公司怎么写周报。

第二个是可重复工作流。每周的营销复盘、客户 call 前的准备、季度业务评审——这些流程不是"知识点"，是肌肉记忆。你做第 10 次的时候，不该再从头教 AI 怎么做第 1 次。

第三个是新能力。创建 PPT、生成 Excel 报告、构建 MCP 服务器——这些能力不需要等下一个大模型版本。通过 Skills，你可以现在就把这些能力"教"给 AI。

这三个维度，就是 Skills 要填补的缺口。

// part 01 · 演变路径

从 Prompt Engineering 到 Skills

01Zero-shot Prompting — "帮我分析这个数据"2023 初
02Few-shot Examples — 在 prompt 里塞几个例子2023 中
03System Prompt Engineering — 角色扮演 + 规则注入2023 末
04Custom Instructions — 可持久化的偏好设置2024 初
05Skills — 可复用、可组合、可分发的知识包2025

如果我们把"怎么让 AI 按你的要求做事"这条演进路径拉出来看，会非常清晰。

最早是Zero-shot Prompting——直接跟 AI 说"帮我分析这个数据"，祈祷它能理解。然后是Few-shot Examples——在 prompt 里塞几个例子，让它模仿。接着是System Prompt Engineering——给 AI 一个角色，注入规则。再到Custom Instructions——至少不用每次都重新说了，偏好可以持久化。

但所有这些方案都有一个根本问题——不可复用、不可组合、不可分发。你精心调教出来的 system prompt，同事要用还得重新复制一遍。

所以第五步就是Skills。一个 Skill 是一个可复用、可组合、可分发的知识包。你写一次，整个团队、甚至整个互联网都可以用。这是质的飞跃，不是量的积累。

// part 01 · 定义

什么是 Agent Skills？

一句话定义

Agent Skills 是一种轻量级、开放的标准格式，用于扩展 AI Agent 的能力和专业知识。一个 Skill 是一个包含指令、脚本、资源和资产的标准化文件夹集合。

类比：如果 Agent 是操作系统，Skill 就是 App；如果 Agent 是人，Skill 就是培训手册。

①
便携性 Portable

同一 Skill 跨 Claude Code、Claude.ai、API、Agent SDK 使用

②
可组合 Composable

品牌 Skill + PPT Skill + 数据 Skill = 完整报告工作流

③
渐进披露 Progressive

元数据始终加载、指令按需加载、资源执行时加载

那 Skills 到底是什么？我给出一个正式的定义。

Agent Skills 是一种轻量级、开放的标准格式，用于扩展 AI Agent 的能力和专业知识。一个 Skill 是一个包含指令、脚本、资源和资产的标准化文件夹集合。

如果用类比来说——如果 Agent 是操作系统，Skill 就是 App；如果 Agent 是一个人，Skill 就是你给他报的培训课程。学完之后，他就能在特定领域做得跟老手一样好。

Skills 有三个核心特性。第一，便携性——同一个 Skill 可以跨平台使用：Claude Code、Claude.ai、API、Agent SDK，甚至其他支持标准的 Agent 产品。

第二，可组合性——品牌 Skill 提供视觉规范、PPT Skill 提供幻灯片能力、数据 Skill 提供查询逻辑，组合起来就是一个完整的报告工作流。这是最强大的地方。

第三，渐进披露——这是技术上的关键创新，下一页我会专门讲。

// part 01 · 核心机制

渐进式披露：三层加载

L1
元数据层 — 始终加载

启动时加载每个 Skill 的 name + description，仅用于判断是否与当前任务相关。上下文成本：~100 tokens / Skill

L2
指令层 — 触发时加载

任务匹配时，完整加载 SKILL.md 的正文内容。建议 < 500 行，提供详细执行逻辑和步骤。

L3
资源层 — 执行中按需加载

脚本、模板、参考文档——只在 Agent 实际执行到某一步时才读取。真正的大块头在这里，但不到用时不占上下文。

渐进式披露是 Skills 最重要的技术创新。它解决了一个核心矛盾——你可能有上百个 Skill，但不能让它们全部塞进上下文窗口。

所以 Skills 设计了三层加载机制。

第一层是元数据。系统启动时，只加载每个 Skill 的名字和描述，大概 100 个 token。这意味着你有 100 个 Skill 也只占 1 万个 token，非常轻量。这层的作用就是让 Agent 知道"我有什么技能可以调用"。

第二层是指令。当 Agent 判断某个 Skill 跟当前任务相关时，才把完整的 SKILL.md 正文加载进来。建议控制在 500 行以内，这样既能提供详细的执行逻辑，又不会撑爆上下文。

第三层是资源。脚本、模板、参考文档——这些才是真正的大块头，但它们只有在 Agent 实际执行到某一步时才读取。比如一个 PDF 处理的 Skill，里面可能有十几个 Python 脚本，但你不处理 PDF 的时候它们完全不会占用上下文。

这个三层设计非常优雅，让 Agent 可以储备大量技能但保持轻量。

// part 01 · 结构

一个 Skill 长什么样？

📁 最简结构

analyzing-marketing-campaign/ ├── SKILL.md └── references/ └── budget_reallocation_rules.md

📁 中等结构

pdf/ ├── SKILL.md ├── forms.md ├── reference.md └── scripts/ ├── check_fillable_fields.py ├── convert_pdf_to_images.py └── extract_form_field_info.py

📁 完整结构

designing-newsletters/ ├── SKILL.md ├── references/ │ └── style-guide.md └── assets/ ├── header.png ├── icons/ └── templates/ ├── newsletter.html └── layout.docx

看一下 Skill 的物理结构。最简的形式——一个 SKILL.md 加一个 references 文件夹，就构成一个完整的 Skill。

中间这个是 PDF 处理的 Skill，它多了一些脚本文件，比如检查可填写字段、PDF 转图片、提取表单信息——这些都是 Python 脚本，放在 scripts 目录里。

右边这个更完整，是设计邮件的 Skill，有 assets 目录放图片和模板。你可以看到，Skills 的复杂度是按需增长的——简单任务只需要一个 SKILL.md，复杂工作流可以包含脚本、模板、参考文档全套。

注意一个关键点：SKILL.md 是唯一必须的文件。它既是元数据容器，也是指令入口。下面我展示一下它长什么样。

// part 01 · 核心文件

SKILL.md 的解剖

--- name: analyzing-marketing-campaign description: Analyze marketing campaign performance data and generate reports. Use when provided with campaign data files or asked to analyze marketing metrics. license: MIT metadata: author: marketing-team version: 2.0.0 --- # Analyzing Marketing Campaign ## Step 1: Data Loading & Validation Load the provided CSV and validate column names against the expected schema... ## Step 2: Metric Computation Calculate CPA, ROAS, CTR using the formulas in references/... ## Step 3: Report Generation Output a formatted Excel report using...
①
YAML Frontmatter — 元数据

name（必填）：小写+连字符，建议 gerund 形式
description（必填）：说明"做什么"和"何时用"

②
Body — Markdown 指令

告诉 Agent 怎么做：步骤、输入格式、输出格式、常见边界情况

③
写作建议

控制在 500 行以内；详细资料移至 references/；引用层级保持一层；描述要具体到能触发自动匹配

④
自由度控制

高自由度：文本指导 → 中自由度：伪代码/模式 → 低自由度：精确脚本

SKILL.md 是 Skill 的灵魂。它分两大块——顶部的 YAML 元数据和下面的 Markdown 指令。

元数据里，name 和 description 是必填的。name 要用小写加连字符，建议用动名词形式，比如 analyzing-marketing-campaign。description 非常关键——它不只说明 Skill 做什么，还要说明何时触发。因为这个 description 就是 Agent 判断要不要调用这个 Skill 的依据。

正文部分就是纯 Markdown，告诉 Agent 怎么做。建议写成步骤形式：第一步做什么、第二步做什么。控制 500 行以内，详细的参考资料放到 references 目录。

右下角这个"自由度控制"值得多说两句。你的 Skill 可以设计成不同粒度——高自由度就是给个方向，AI 自己发挥；中自由度是给伪代码和模式，有个参考但允许变化；低自由度就是精确脚本，必须严格按流程走。具体用哪种，取决于你的场景对一致性的要求。

// part 02 · 开放标准

为什么开放标准是关键？

🔒 没有标准的世界

每个 AI 产品自己做一套"插件系统"：
• A 平台的技能不能在 B 平台用
• 开发者要为每个平台各写一份
• 用户被锁定在单一生态

🔓 有标准的世界

一个 Skill 文件夹，所有平台通用：
• 开发者写一次，到处运行
• 用户换平台不丢失技能
• 社区贡献复利增长

agentskills.io 规范核心

① 技能目录结构标准  ② SKILL.md 元数据标准  ③ 技能接口协议  ④ 安全审计标准  ⑤ 发布与版本管理

为什么开放标准这么重要？我做一个对比就很清楚了。

左边是没有标准的世界——每个 AI 产品自己做一套"插件系统"。你在 Claude 上写的技能，在 Cursor 上用不了；开发者要为每个平台各写一份；用户一旦用了某个平台就被锁死了。

右边是有标准的世界——一个 Skill 文件夹，所有支持标准的平台通用。开发者写一次到处运行，用户换平台不丢失技能，社区贡献复利增长。

agentskills.io 定义了五个核心规范：目录结构标准、元数据标准、接口协议、安全审计标准、发布与版本管理。由 Anthropic 牵头，联合数十家头部企业共同制定。这是一个真正意义上的行业公共品。

// part 02 · 定位辨析

Skills vs MCP vs Tools vs Subagents

MCP
连接外部系统和数据

MCP Server 连接数据库 → 提供"访问"。Skills 说"用 A 列和 B 列计算指标 X" → 提供"专知"。

Tools
提供 Agent 基础能力

Tool 定义始终占上下文；Skills 按需加载。Tool 能力固定；Skills 可以包含"工具即需"的脚本。

Sub
独立隔离的子 Agent

Subagent 有自己的上下文和权限；Skills 为主 Agent 和所有子 Agent提供专业知识。

很多人会混淆 Skills、MCP、Tools 和 Subagents。我来做一个清晰的辨析。

MCP 解决的是"连接"问题——连接你的数据库、API、服务。一个 MCP Server 把数据库接通了，但数据库里的数据怎么用、算什么指标，MCP 不管。这就是 Skills 要做的事——MCP 提供访问，Skills 提供专知。

Tools 是 Agent 的基础能力——但 Tool 定义始终占上下文，而 Skills 是按需加载的。而且 Tools 的能力是固定的，Skills 可以包含"工具即需"的脚本，需要的时候才编译成工具。

Subagents 是独立隔离的子 Agent——它们有自己的上下文和权限。而 Skills 是知识层，可以同时为主 Agent 和所有子 Agent 提供专业知识。在 Claude Code 里，你甚至可以在子 Agent 定义里指定它使用哪些 Skills。

一句话总结：MCP 接通路，Tools 给手，Subagents 分工，Skills 给脑子。

🔑 关键设计决策

1. 为什么选择开放标准？
避免生态锁定，复用社区成果

2. 为什么三级存储？
用户级跨项目复用，项目级团队共享，内置保障基线

3. 为什么双重触发？
新用户零学习成本（自动），老用户精确控制（手动）

WorkBuddy 的技能架构有几个关键设计决策，我一一说明。

第一，我们完全兼容 Agent Skills 开放标准。SKILL.md 驱动、渐进式披露、跨平台可移植。为什么？因为我们要避免生态锁定，而且可以复用社区的成果——anthropics/skills 仓库里那些高质量的 Skill，WorkBuddy 用户可以直接用。

第二，三级技能存储。用户级存在 ~/.workbuddy/skills/，跨项目复用；项目级存在 .workbuddy/skills/，团队共享；还有内置技能保障基线能力。

第三，双重触发机制。自动触发通过关键词匹配智能调用——你说"画一个架构图"，系统自动加载 diagram Skill。手动触发用 @skill:name 显式指定——你明确知道自己要用什么。这个设计对新用户零学习成本，对老用户提供精确控制。

第四，MCP 和 Skills 双层增强。MCP 连接外部系统，Skills 提供领域专知——两者协同而非替代。

// part 04 · 创建路径

WorkBuddy 的三种技能创建方式

🎨 路径一：图形化创建

在 Skills 管理界面可视化配置，无需写代码。定义触发词、执行动作和规则。推荐新手

📝 路径二：YAML 模板导入

基于标准 YAML 语法编写 Skill 定义文件，一键导入。适合复杂逻辑和团队协作。

🤖 路径三：AI 反向生成

当你执行三次结构相似的操作，系统自动识别模式并主动提示生成 Skill。最具"AI 感"的方式。

💡 实战心得

路径三的反向生成是最被低估的功能。很多用户不知道——他们重复操作 3 次后，系统会弹窗提示"检测到重复模式，是否创建为技能？"。这是 Skills 从"被动工具"变成"主动助手"的关键转折。

WorkBuddy 提供了三种创建 Skill 的方式，对应不同技术背景的用户。

路径一是图形化创建，在管理界面可视化配置，定义触发词、执行动作和规则。完全不用写代码，推荐新手用。

路径二是 YAML 模板导入。用标准的 YAML 语法写一个 Skill 定义文件，一键导入。适合复杂逻辑和需要团队协作的场景——YAML 天然就支持版本管理。

路径三最有意思——AI 反向生成。当你在 WorkBuddy 里执行了三次结构相似的操作，比如三次把不同路径的 Word 转成 PDF，系统会智能识别这个模式，主动提示你"是否创建为技能？"。然后它自动生成一个带变量参数的 Skill。

这个反向生成是最被低估的功能。它让 Skills 从"被动工具"——你得主动去创建——变成了"主动助手"——它自己发现你的重复模式并帮你自动化。这是真正体现 AI 价值的设计。

// part 04 · 踩坑

我们踩过的坑

❌ 坑 1：Description 写得太泛

"Helps with documents" → Agent 不知道何时该触发。正确写法："Create, read, and edit Word documents (.docx). Use when user provides a .docx file or asks to create a document."

❌ 坑 2：SKILL.md 超过 1000 行

指令太长，Agent 抓不住重点，执行效果反而更差。解决：拆分成主指令 + references，保持 500 行以内。

❌ 坑 3：忽视渐进式披露

把所有参考资料都塞进 SKILL.md，导致一触发就吃掉大量上下文。解决：L1/L2/L3 三层分离。

❌ 坑 4：Skill 之间职责重叠

两个 Skill 都在处理文档格式，Agent 不知道该用哪个。解决：description 里明确触发条件边界。

开发 Skills 的过程中我们也踩了不少坑。我挑四个最典型的分享。

第一个坑——Description 写得太泛。"Helps with documents" 这种描述，Agent 完全不知道什么时候该触发。正确的写法应该是具体到什么文件类型、什么用户行为触发。比如 "Create, read, and edit Word documents. Use when user provides a .docx file or asks to create a document."

第二个坑——SKILL.md 超过 1000 行。很多人觉得指令越详细越好，但实际上太长了 Agent 抓不住重点，执行效果反而更差。我们的经验是控制在 500 行以内，详细的参考资料移到 references 目录。

第三个坑——忽视渐进式披露。有人把所有参考资料都塞进 SKILL.md，一触发就吃掉大量上下文。解决方案就是严格执行 L1/L2/L3 三层分离。

第四个坑——Skill 之间职责重叠。两个 Skill 都在处理文档格式，Agent 不知道该用哪个。解决方法是在 description 里明确触发条件的边界。

// part 04 · 最佳实践

技能开发最佳实践

01
Description 决定生死

同时说明做什么和何时用，包含具体关键词。这是 Agent 自动匹配的唯一依据。

02
指令要像 SOP，不要像散文

步骤化、结构化、可验证。每一步都可以判断"做对了没有"，而不是开放式的指导。

03
References 是"字典"，不是"小说"

每个 reference 文件聚焦一个主题，超过 100 行要加目录。Agent 需要的是可查可引用的知识块，不是长篇大论。

04
测试所有目标模型

Sonnet 和 Haiku 的理解能力不同。同一个 Skill 在不同模型上表现可能差异很大，务必实际测试。

总结一下技能开发的最佳实践，四条。

第一条，Description 决定生死。它是 Agent 自动匹配的唯一依据。你必须同时说明"做什么"和"何时用"，包含具体关键词。写得好的 description，可以让 Agent 在用户说出相关词汇的时候百分之百自动触发。

第二条，指令要像 SOP，不要像散文。步骤化、结构化、可验证。每一步都应该可以判断"做对了没有"——像标准操作流程，不是开放式的指导。

第三条，References 是字典，不是小说。每个 reference 文件聚焦一个主题。超过 100 行的要加目录，方便 Agent 快速定位。它需要的是可查可引用的知识块，不是长篇大论。

第四条，测试所有目标模型。Sonnet 和 Haiku 的理解能力不同，同一个 Skill 在不同模型上表现可能差异很大。务必实际测试，不能只在你的开发模型上验证过就上线。

趋势一：技能市场生态化

更多高质量 Skill → 更多用户采用
→ 更多开发者贡献 → 更多高质量 Skill

类比：App Store 2008 年启动时的飞轮——一旦越过临界点，增长将呈指数级。

第一个趋势——技能市场生态化。

有预测说到 2027 年，全球 Agent 技能市场规模会超过 1000 亿美元。这个数字我不保证准确性，但方向是对的。会出现专门的技能开发公司，就像现在有专门的 App 开发公司一样。法律、医疗、金融这些垂直领域，会有专业的技能供应商。

更关键的是飞轮效应——更多高质量 Skill 吸引更多用户，更多用户吸引更多开发者，更多开发者产出更多高质量 Skill。这跟 2008 年 App Store 刚启动时的逻辑一模一样。一旦越过临界点，增长将呈指数级。

我们现在可能正处于那个临界点附近。

// part 05 · 趋势2

趋势二：自进化技能

①
基于使用反馈自动优化

用户修改了 AI 输出的某些部分 → 系统识别差异 → 自动更新 Skill 的 references

②
跨用户知识蒸馏

100 个用户都在同一类任务上做了相似修改 → 蒸馏成通用最佳实践 → 推送到所有用户的 Skill

③
版本管理与回滚

每次自动更新都带版本号，效果变差可以一键回滚。Skill 和代码一样需要 CI/CD。

🧬 本质变化

今天的 Skill 是静态的知识包——写好就不会变，除非手动更新。

明天的 Skill 是活的有机体——它会随着使用不断进化，越用越好，越用越懂你。

这是从"软件"到"生命体"的质变。

第二个趋势——自进化技能。这是我最兴奋的一个方向。

今天的 Skill 是静态的知识包——你写好了就不再变，除非手动更新。但未来会不同。

第一层，基于使用反馈自动优化。如果用户每次都在 AI 输出的某些部分做相同的修改，系统能识别出这个差异，自动更新 Skill 的参考信息。你不用自己改，Skill 自己学会了。

第二层，跨用户知识蒸馏。100 个用户都在同一类任务上做了相似修改，系统把这种共性蒸馏出来，变成通用最佳实践，推送给所有使用这个 Skill 的用户。这就像集体智慧在自动汇聚。

第三层，版本管理和回滚。自动更新不总是对的方向，需要带版本号，效果变差就一键回滚。Skill 和代码一样需要 CI/CD。

从静态知识包到活的有机体——这是从"软件"到"生命体"的质变。

终极桥梁

过去
知识 → Prompt → 不可靠的输出

现在
知识 → Skill → 可复用的行动

知识被编码成 Skill，按需加载、可组合、可分发。知识是"活的"，传递是"永久性"的。

未来
知识 → 自进化 Skill → 持续优化的行动

Skill 在使用中自动优化、跨用户蒸馏最佳实践。知识变成"会生长的"，行动变成"越来越好的"。

第三个趋势，也是我今天最想传达的核心观点——Skills 是从知识到行动的终极桥梁。

过去，知识通过 prompt 传递给 AI，每次都要重新描述，输出质量波动大。知识是"死的"，传递是"一次性"的。

现在，知识被编码成 Skill，按需加载、可组合、可分发。知识变成"活的"，传递变成"永久性"的。你写一次，所有人都能用。

未来，Skill 在使用中自动优化，跨用户蒸馏最佳实践。知识变成"会生长的"，行动变成"越来越好的"。

这三步演化，本质上是知识的复利效应——每一次使用都在为下一次使用积累价值。这才是 Skills 真正的终局——不是"教 AI 做事"，而是"让知识在 AI 生态中流动和进化"。

// 核心洞察

它是 知识可组合化、可分发化、可进化化

我再说一遍这个核心洞察——Skills 不只是"教 AI 做事"，虽然这是最直观的理解。

更深层的是，Skills 是知识可组合化、可分发化、可进化化的基础设施。

可组合——品牌 Skill 加 PPT Skill 加数据 Skill，组合出完整的报告工作流。
可分发——你写好一个 Skill，同事、社区、整个互联网都能用。
可进化——Skill 在使用中自动优化，越用越好。

这三点加在一起，让 Skills 成为 AI Agent 生态的基础设施层。就像 HTTP 是互联网的基础设施、App 是智能手机的基础设施一样，Skills 会成为 AI Agent 的基础设施。
